# The Anchor Asset Framework

*(A repeatable system for finishing useful things without burnout)*

---

## 0. First Principles (Do Not Skip)

* I am a **systems builder**, not a maintainer.
* Functional completion is **real completion**.
* Polish is optional unless it prevents harm.
* Ownership creates burnout; infrastructure does not.
* My goal is **durable independence**, not expression.

If a project violates any of these, it is the wrong shape.

---

## 1. Problem Qualification (Is this worth building?)

Only proceed if **all** are true:

* I personally experience this problem
* It recurs more than once
* Existing solutions annoy me
* The solution can exist without me
* The problem still matters if I disappear for months

If any answer is “no” → stop.

---

## 2. Project Shape (Non-Negotiable)

The project **must** be:

* Tool, script, or utility
* Local-first or low-touch
* Complete when it works
* Usable without explanation
* Able to be abandoned safely

If it requires:

* Continuous care
* Community
* Narrative
* Ongoing motivation

→ reject it.

---

## 3. Build Phase (The Only Creative Phase)

Rules:

* Solve the problem **only for myself**
* No roadmap
* No feature wishlist
* No audience thinking
* No branding
* No UI perfection

Stop building when:

> “This removes the frustration from my life.”

That is the **finish line**, not the midpoint.

---

## 4. Stabilization Phase (Short, Boring, Finite)

Do only the following:

* Add sane defaults
* Remove sharp edges
* Make failure modes obvious
* Ensure repeatability
* Write minimal documentation

**No expansion. No elegance. No optimization.**

This phase should feel boring — that’s the signal to stop.

---

## 5. The Integrity Pass (Exactly One)

You are allowed **one** pass to protect standards.

Answer only these questions:

1. Is anything actively sloppy?
2. Is anything misleading or dishonest?
3. Would a competent peer be confused or embarrassed using this?

If all answers are “no” → freeze the project.

No second pass.

---

## 6. Release Rule (Let Go of Ownership)

Once the integrity pass is complete:

* The project no longer represents me
* It is infrastructure, not identity
* Silence is success
* Improvement is optional
* Maintenance only occurs if *I* need it

At this point, I am allowed to walk away.

---

## 7. Monetization (Optional, Low-Friction)

Monetization is allowed **only if** it does not change the project shape.

Acceptable:

* One-time purchase
* Simple checkout
* Honor-based or lightweight licensing
* Minimal description

Not acceptable:

* Funnels
* Growth tactics
* Repositioning
* Emotional attachment to results

Money is a side effect, not validation.

---

## 8. Success Criteria (Binary)

A project is successful if:

* It works
* It stands on its own
* I do not feel obligated to it
* It continues without attention

Revenue is a bonus, not the definition.

---

## 9. Long-Term Strategy (How this compounds)

* Build one asset at a time
* Finish it completely
* Let it run
* Stack outcomes, not effort
* Never maintain more than necessary

The sequence is long-term.
Each project is finite.

---

## 10. Final Guardrail (Read This When Stuck)

> “If it already works and I am still polishing, I am protecting identity, not improving value.”

That is the signal to stop.

---
